# Auto-apply VK Call Link

> Бот автоматически применяет ссылку на VK-звонок: сохраняет, публикует в VK, проверяет TURN и отчитывается.

## Архитектура (контекст)

```
VPS:    vk-turn-proxy-server  — UDP relay, ссылку не знает
        vk-turn-bot           — Telegram бот (этот проект)

Роутер: vk-turn-proxy-client  — потребляет ссылку, TURN-сессия

Канал:  закреплённый пост в закрытой VK-группе
        бот → wall.edit → роутер (крон) → wall.get → перезапуск клиента
```

- Сервер **не** перезапускается при смене ссылки.
- Бот публикует ссылку в VK; роутер сам забирает и перезапускает клиент.

## Flow: applyLink

```typescript
async function applyLink(link: string, ctx: Context): Promise<void>
```

Вызывается из `/setlink <url>` и при получении VK-ссылки в тексте сообщения.

**Concurrency guard:** если `applyLink` уже выполняется — отвечаем "Уже применяю ссылку, подожди" и выходим. Реализация: boolean-флаг `isApplying`.

**Дедупликация:** если `link === config.vkCallLink` — отвечаем "Эта ссылка уже активна" и выходим.

```
Ссылка приходит (через /setlink или просто текстом)
  |
0. Concurrency guard + дедупликация
  |
1. Валидация формата (regex vk.com/call/join/...)
  |
2. Сохранение в /etc/vk-turn-proxy/config.json
  |
3. Публикация в VK (wall.edit закреплённого поста)
   |- VK-креды есть -> публикуем -> OK
   |- VK-кредов нет -> пропускаем -> "VK relay не настроен"
  |
4. Health check (до 5 попыток с интервалом 10 сек)
   Запускаем vk-turn-client с таймаутом, проверяем TURN
  |
5. Отчёт в Telegram
```

### Шаги

| Шаг | Действие | При ошибке |
|-----|----------|------------|
| 0 | Concurrency guard + дедупликация | "Уже применяю" / "Ссылка уже активна" |
| 1 | Валидация regex `^https://vk\.com/call/join/[A-Za-z0-9_/-]+$` | Отказ, сообщение пользователю |
| 2 | Сохранение link + updatedAt в config.json | Отказ: "Не удалось сохранить конфиг: {error}" |
| 3 | `wall.edit` закреплённого поста через VK API | Warning, продолжаем (VK опционален) |
| 4 | Health check: до 5 попыток, интервал 10 сек | После 5 неудач — warning |
| 5 | Финальный отчёт в Telegram | — |

### Health check (retry-версия)

Упрощённая проверка — только TURN-клиент (без systemd/UDP, т.к. проверяем валидность ссылки, а не состояние сервера):

```bash
timeout 15 ${VK_TURN_CLIENT_PATH} \
  -link "<url>" \
  -peer ${config.wgPeerAddress}:${config.turnListenPort} \
  -listen 127.0.0.1:0 \
  -n 1
```

- До 5 попыток, интервал 10 сек между ними.
- Успех = клиент подключился (или таймаут > 15 сек, значит сессия установлена — клиент при невалидной ссылке завершается с ошибкой быстро, до таймаута).
- Неудача = клиент вернул ошибку (exit code != 0) до таймаута.

**Scope:** проверка подтверждает, что VK call link валидна и TURN-инфраструктура VK доступна с VPS. Она **не** подтверждает, что роутер забрал новую ссылку — это зависит от крона роутера (до 5 мин).

### VK API

Один HTTP-вызов, без внешних зависимостей (native `fetch`):

```
POST https://api.vk.com/method/wall.edit
  access_token = VK_GROUP_TOKEN
  owner_id     = -VK_GROUP_ID
  post_id      = VK_POST_ID
  message      = <link>
  v            = 5.199
```

**Ответ (успех):** `{"response": {"post_id": 123}}` — проверяем наличие `response.post_id`.

**Ответ (ошибка):** `{"error": {"error_code": N, "error_msg": "..."}}` — логируем `error_code` + `error_msg`, показываем warning. Все VK-ошибки обрабатываются одинаково (warning + продолжаем), т.к. VK опционален.

**Конфигурация VK:** все три переменные (`VK_GROUP_TOKEN`, `VK_GROUP_ID`, `VK_POST_ID`) должны быть заданы. Если задана только часть — считаем VK не настроенным, логируем warning при старте.

### Прогресс-сообщения

Бот отправляет одно сообщение и редактирует его по мере выполнения:

```
[1] Применяю ссылку...
[2] Ссылка сохранена. Публикую в VK...
[3] VK обновлён. Проверяю TURN (1/5)...
[4] Проверяю TURN (2/5)...
[5] Ссылка применена, TURN жив, VK обновлён
```

Варианты финального отчёта:
- `Ссылка применена, TURN жив, VK обновлён`
- `Ссылка применена, TURN жив, VK relay не настроен`
- `Ссылка сохранена, VK обновлён, но TURN не отвечает (5/5 попыток)`
- `Ссылка сохранена, но TURN не отвечает (5/5), VK relay не настроен`

## Изменения в командах

### `/setlink <url>` — рефакторинг
- Вызывает `applyLink(link, ctx)` вместо текущей `updateCallLink` + "выполни /restart"

### Текстовое сообщение с VK-ссылкой
- Вместо "Похоже на ссылку, применить?" — сразу вызывает `applyLink(link, ctx)`

### `/restart` — упрощение
- Просто `systemctl restart vk-turn-proxy`
- Ждёт 3 сек, проверяет `systemctl is-active`
- Инкрементирует `stats.totalRestarts`
- Без манипуляций с systemd override (сервер — статичный UDP relay, аргументы не меняются)

### `/status`, `/stats`, `/config` — без изменений

### Фоновый мониторинг
- Без изменений, использует полный `checkTurnHealth()` (systemd + UDP + клиент)
- При алерте подсказывает: "Отправь новую ссылку VK-звонка"

## Удаляемый код

- Логика systemd override в `restartService()` (строки 173-185): сервер — статичный relay, аргументы запуска не меняются при смене ссылки
- `updateCallLink()` целиком — логика поглощена `applyLink()`
- Предложение "/setlink" в обработчике текстовых сообщений (строки 361-366)

## Новые env-переменные

```env
VK_GROUP_TOKEN=     # токен сообщества с правами wall
VK_GROUP_ID=        # ID группы (без минуса)
VK_POST_ID=         # ID закреплённого поста
```

Все три обязательны для работы VK relay. Если задана только часть — VK relay считается не настроенным.

## Зависимости

Новых npm-зависимостей нет. VK API вызывается через native `fetch` (Node 20).
