# Link Queue & Fallback Design

## Проблема

Бот хранит одну активную ссылку. Если она умирает — нужно ручное вмешательство. Нет резерва.

## Решение

Очередь резервных ссылок с автоматическим fallback при смерти активной.

---

## 1. Модель данных

Добавить поле `linkQueue: string[]` в `TurnConfig`:

```typescript
interface TurnConfig {
  // ... существующие поля ...
  linkQueue: string[];  // массив резервных VK call ссылок
}
```

- `vkCallLink` — текущая активная ссылка (без изменений)
- `linkQueue` — резервный массив
- Обратная совместимость: при загрузке конфига, если `linkQueue` отсутствует — инициализировать как `[]`

## 2. Поведение существующих команд

### `/setlink <url>` — принудительная установка

- Валидация ссылки (существующий regex)
- Ссылка становится активной `vkCallLink`
- Очередь `linkQueue` не трогается
- Остальной flow без изменений: сохранить → VK пост → health-check

### Автодетект ссылки в чате — добавление в очередь

**Было:** автодетект сразу применяет ссылку (вызывает `applyLink`).

**Стало:**
- Валидация ссылки (строгий `VK_CALL_REGEX` с `$` и `[A-Za-z0-9_/-]+$`)
- Проверка дубликатов: если ссылка уже в очереди или является активной — сообщить и не добавлять
- Добавить в конец `linkQueue`, сохранить конфиг
- Ответить: "Ссылка добавлена в очередь (позиция N). Всего ссылок: M"
- **Не** активировать, **не** постить в VK, **не** проверять здоровье

## 3. Fallback-логика

### Когда срабатывает

- При старте бота (после `bot.start()`, до отправки стартового сообщения)
- При каждом фоновом health-check, если активная ссылка мертва

### Concurrency

Fallback sweep проверяет флаг `isApplying` перед началом. Если `isApplying === true` (например, идёт `/setlink`) — sweep пропускается, будет выполнен при следующем health-check цикле. Аналогично, во время fallback sweep `isApplying` установлен в `true`, поэтому `/setlink` ответит "Уже применяю" (существующее поведение).

### Алгоритм

Для проверки ссылок при fallback используется `checkLinkHealth()` напрямую (не `checkTurnHealth()`), с уменьшенным числом попыток (1-2 вместо 5), чтобы sweep не занимал минуты.

```
1. Проверить активную vkCallLink через checkLinkHealth(maxAttempts=2)
2. Если ОК → ничего не делать
3. Если FAIL → перебрать linkQueue по порядку (с индекса 0):
   - Пропустить ссылки, равные текущей vkCallLink
   a. checkLinkHealth(queue[i], maxAttempts=2)
   b. Если ОК:
      - Установить queue[i] как активную vkCallLink
      - Опубликовать в VK (если VK пост не удался — не фатально, ссылка всё равно активирована)
      - Уведомить админов: "Ссылка умерла, переключился на резервную (N осталось в очереди)"
      - Остановить перебор
   c. Если FAIL:
      - Удалить queue[i] из массива
      - Уведомить админов: "Резервная ссылка тоже мертва, удалена"
      - Перейти к следующей
4. Если очередь исчерпана:
   - Уведомить админов: "Все ссылки мертвы, валидных ссылок нет!"
```

При переключении на резервную — ссылка остаётся в очереди. Удаляется только при провале health-check. При итерации очереди ссылка, равная текущей `vkCallLink`, пропускается.

## 4. Новые команды

### `/links` — просмотр очереди

Вывод:

```
🔗 Активная: https://vk.ru/call/join/abc123

Очередь (3):
1. https://vk.ru/call/join/def456
2. https://vk.ru/call/join/ghi789
3. https://vk.com/call/join/jkl012
```

Если очередь пуста: "Очередь пуста, резервных ссылок нет".

### `/rmlink N` — удаление из очереди

- N — номер из списка `/links` (начиная с 1)
- Валидация: число, в пределах массива
- Удалить, сохранить конфиг
- Ответить: "Ссылка #N удалена. Осталось в очереди: M"

Кнопки в клавиатуре не добавляются — команды используются редко.

## 5. Нотификации и отчёты

### При старте бота

К существующему сообщению о запуске добавить:
- "Ссылок в очереди: N" (или "Очередь пуста" если 0)

### В ежедневном отчёте (04:00 UTC)

Добавить строку:
- "Ссылок в очереди: N"

Без валидации ссылок — просто `linkQueue.length`. Реальная валидация происходит только при fallback.

## 6. Рандомизированный интервал health-check

### Было

Фиксированный `setInterval` каждые `CHECK_INTERVAL_MIN` минут.

### Стало

Цепочка `setTimeout`: каждый health-check по завершении планирует следующий с новым случайным интервалом.

```typescript
function randomCheckInterval(): number {
  const minMs = 5 * 60 * 1000;   // 300000
  const maxMs = 10 * 60 * 1000;  // 600000
  let ms = minMs + Math.floor(Math.random() * (maxMs - minMs + 1));
  // Избегаем круглых значений (кратных 30000); корректируем в пределах диапазона
  while (ms % 30000 === 0) {
    ms = minMs + Math.floor(Math.random() * (maxMs - minMs + 1));
  }
  return ms;
}
```

`CHECK_INTERVAL_MIN` остаётся в `.env` но не используется.

### Graceful shutdown

При остановке бота — очищать текущий `setTimeout` handle (аналогично существующему `clearInterval(monitorInterval)`).

## 7. Сводка изменений

| Компонент | Изменение |
|---|---|
| `TurnConfig` | + `linkQueue: string[]` |
| Автодетект ссылки | Добавляет в очередь, не активирует |
| `/setlink` | Без изменений, очередь не трогает |
| `/links` | Новая — показать активную + пронумерованную очередь |
| `/rmlink N` | Новая — удалить из очереди по номеру |
| Health-check fallback | При смерти активной — перебор очереди, мёртвые удаляются |
| Старт бота | Fallback + нотификация "ссылок в очереди: N" |
| Ежедневный отчёт | + строка "ссылок в очереди: N" |
| Интервал проверок | `setTimeout` с рандомом 5-10 мин, без круглых значений |
| `CHECK_INTERVAL_MIN` | Остаётся в .env, не используется |
